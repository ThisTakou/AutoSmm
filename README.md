# AutoSmm для FunPayCardinal

Переработанная версия плагина авто-накрутки для `FunPayCardinal`.

Плагин принимает ссылку от покупателя, создает заказ через SMM API, отслеживает статус и позволяет работать с заказами прямо из FunPay-диалога и Telegram-панели.

## Что умеет

- создание заказов через SMM API;
- работа с двумя API;
- отслеживание статусов;
- рефилл;
- уведомления в Telegram;
- дробление заказа, если сервис не принимает весь объем сразу;
- управление через `/autosmm`.

## Установка

1. Поместите файл плагина в папку `plugins` вашего `FunPayCardinal`.
2. Перезапустите бота.
3. Откройте `/autosmm`.
4. Укажите API-настройки.

Если раньше у вас уже стояла другая версия `AutoSmm`, лучше оставить в папке `plugins` только один активный файл плагина, чтобы не было конфликтов.

## Telegram-команды

- `/autosmm` — открыть настройки плагина
- `/check_balance` — проверить баланс SMM

## Команды для покупателя

- `#статус ID` — проверить статус заказа
- `#рефилл ID` — запросить рефилл

## Как работает дробление заказа

Некоторые SMM-сервисы не дают создать один большой заказ, если количество выходит за лимит конкретной услуги. Обычно API в таком случае возвращает ошибку вроде ограничения по `min/max` количеству.

В этой версии добавлено автоматическое дробление:
- если сервис не принимает весь объем целиком, плагин пытается определить лимит;
- после этого заказ делится на несколько частей;
- каждая часть создается отдельно;
- покупатель получает информацию по созданным частям;
- дальше можно отслеживать статус каждой части отдельно.

Это позволяет не терять заказ целиком из-за лимита услуги и не делать всё вручную.

## Примечание

Плагин не является отдельной системой накрутки — он только работает как связка между `FunPayCardinal` и вашим SMM API.

Перед использованием лучше проверить:
- правильность `API URL`;
- правильность `API KEY`;
- лимиты услуг на стороне SMM-панели;
- доступность рефилла для нужных услуг.

## Как работает дробление заказа

Если SMM-сервис не принимает весь объем одним заказом из-за лимита, плагин не останавливается на ошибке, а пытается разбить заказ на несколько частей.

Что происходит:
- покупатель отправляет ссылку и подтверждает заказ;
- плагин пробует создать заказ на весь объем;
- если API отвечает ограничением по количеству, плагин определяет допустимый лимит;
- после этого общий объем делится на несколько отдельных частей;
- каждая часть отправляется в SMM API отдельно;
- если части удалось создать, покупатель получает сообщение с ID созданных частей;
- дальше покупатель может проверять статус каждой части по отдельности.

Что именно видит покупатель:
- сообщение о том, что заказ был разделен;
- общее количество;
- сколько частей удалось создать;
- список ID по каждой части;
- команды для проверки статуса и рефилла.

Пример:
- заказ на 5000 единиц не проходит целиком;
- API разрешает максимум 1000;
- плагин делит заказ на 5 частей по 1000;
- создает их отдельно;
- отправляет покупателю список созданных частей.

Если часть заказов создать не удалось, плагин сообщает об этом отдельно.
